<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Astro page props -> Head.astro attribute injection | Astro auto-escapes `{expr}` in attributes, safe by default |
| Astro build-time JS object -> JSON-LD `<script>` via `set:html` | `set:html` bypasses HTML escaping; content must be a hand-maintained literal (no runtime input) |
| Public web -> /robots.txt | Read-only static file, no injection surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-7-01 | Tampering/XSS | `Head.astro` JSON-LD `set:html` injection point | mitigate | All JSON-LD values are hand-maintained literals in the frontmatter; add inline comment documenting the discipline; never interpolate `Astro.props` values into `jsonLd` without escaping `</` as `<\/`. Additionally, `og:image:alt` uses `content={ogImageAltText}` (Astro auto-escapes — safe). |
| T-7-06 | Info Disclosure (allowlist bypass) | `ogImageSlug` prop value flows into the `og:image` URL `/og/<slug>.png` | accept (in this plan) | This plan only wires the prop to an href; the actual slug allowlist + 404 response lives in Plan 04's `[slug].png.ts` endpoint. Head.astro trusts whatever slug the page passes; page code is hand-authored (not user input). |

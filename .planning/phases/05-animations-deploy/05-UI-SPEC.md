## Animation Contracts

This is the primary new design surface in Phase 5. All contracts specified here are definitive.

### Scroll-Reveal System

**Implementation:** CSS scroll-driven animations via `animation-timeline: view()` — zero JavaScript. Applied to static `.astro` section components via utility classes in `global.css`.

**Hero excluded:** HeroSection already animates via Motion v12 on load. Do not add scroll-reveal to `#hero`.

#### Keyframe Definition

```css
@keyframes reveal-up {
  from {
    opacity: 0;
    transform: translateY(16px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}
```

Travel distance: 16px (`translateY(16px)`) — midpoint of D-02 range (15–20px). Matches hero entrance y-offset of 20px closely enough to feel consistent without being identical.

#### Base Reveal Class

```css
.reveal {
  animation: reveal-up 0.5s ease-out both;
  animation-timeline: view();
  animation-range: entry 0% entry 40%;
}
```

Duration: 0.5s — midpoint of D-02 range (0.4–0.6s).
Easing: `ease-out` — decelerating entrance, matches hero's custom cubic-bezier feel at macro level.
`animation-fill-mode: both` — element stays at `opacity: 0` before entering viewport; locks at `opacity: 1` after.
`animation-range: entry 0% entry 40%` — animation completes by the time 40% of the element has entered the viewport, so content is always legible when it comes to rest.

#### Staggered Children

For sections with multiple child elements (timeline items, skill badges, project card elements), use delay offsets:

```css
.reveal-stagger > * {
  animation: reveal-up 0.5s ease-out both;
  animation-timeline: view();
  animation-range: entry 0% entry 50%;
}

.reveal-stagger > *:nth-child(1) { animation-delay: 0ms; }
.reveal-stagger > *:nth-child(2) { animation-delay: 80ms; }
.reveal-stagger > *:nth-child(3) { animation-delay: 160ms; }
.reveal-stagger > *:nth-child(4) { animation-delay: 240ms; }
.reveal-stagger > *:nth-child(5) { animation-delay: 320ms; }
.reveal-stagger > *:nth-child(6) { animation-delay: 400ms; }
.reveal-stagger > *:nth-child(n+7) { animation-delay: 400ms; } /* cap at 6 steps */
```

Stagger delay: 80ms per child — compressed from hero's 150ms because scroll-driven animations have variable trigger timing; tighter stagger ensures all children complete before user scrolls past.

#### Per-Section Application

| Section | Component | Reveal Strategy | Class Applied To |
|---------|-----------|-----------------|-----------------|
| About | `AboutSection.astro` | Single reveal — section intro block | `.reveal` on the `<section>` or inner prose container |
| Experience | `ExperienceTimeline.astro` | Staggered children — each timeline item | `.reveal-stagger` on timeline list; each `<li>` item is a stagger child |
| Skills | `SkillsGrid.astro` | Staggered children — each category block | `.reveal-stagger` on skills grid container; each category div is a stagger child |
| CV Download | `CVDownloadButton.astro` | Single reveal | `.reveal` on button wrapper |
| Projects | `ProjectsSection.astro` | Single reveal — entire section enters together | `.reveal` on `<section>` inner container |
| Contact | `ContactSection.astro` | Single reveal — heading + links block | `.reveal` on the content wrapper |

**Rationale for group-vs-stagger decisions:**
- Experience timeline: multiple discrete entries warrant individual stagger — each role is a reading unit
- Skills grid: each category (Languages, Frameworks, etc.) is a discrete unit — stagger aids scanning
- Projects: single card now — no stagger needed; structure ready for stagger when second card added
- Contact: 3 links, but contact section is a CTA block — group entrance feels more decisive than stagger

### Reduced Motion

Extends existing `@media (prefers-reduced-motion: reduce)` block in `global.css`:

```css
@media (prefers-reduced-motion: reduce) {
  .reveal,
  .reveal-stagger > * {
    animation: none;
    opacity: 1;
    transform: none;
  }
}
```

Content renders immediately at full opacity with no animation. No polyfill, no fallback JavaScript. Source: CONTEXT.md D-06.

### Browser Compatibility Fallback

`animation-timeline: view()` is supported in Chrome 115+, Safari 18+, Edge 115+. Firefox as of April 2026 does not support scroll-driven animations.

Fallback behavior: browsers that do not support `animation-timeline` ignore the property and render content at whatever `opacity` and `transform` the non-animated state produces. Because `animation-fill-mode: both` is applied, unsupported browsers that parse the animation but not the timeline will show the `from` state (opacity: 0) permanently.

**Required safe fallback rule in `global.css`:**

```css
/* Safe base state — ensures content is visible in non-supporting browsers */
@supports not (animation-timeline: view()) {
  .reveal,
  .reveal-stagger > * {
    opacity: 1;
    transform: none;
    animation: none;
  }
}
```

This satisfies CONTEXT.md D-05: content shows immediately in Firefox and any other non-supporting browser. Source: REQUIREMENTS.md Phase 5 Success Criterion 2.
